> **이 규칙은 Rigid — 상황에 맞게 변형하지 마라.**

# TDD Rule

## 0. 프로젝트별 규칙 우선 조항

이 파일은 **기본값(fallback)**이다. 프로젝트의 `CLAUDE.md` 또는 `AGENTS.md`에 TDD 관련 규칙이 존재하면 **그 규칙이 이 파일보다 우선**한다. 충돌 시 프로젝트 규칙을 따르고, 이 파일을 참고용으로만 사용하라.

---

## 1. 적용 기준

### TDD 필수 적용 대상

- **비즈니스 로직** — 조건 분기, 계산, 상태 전이
- **API 엔드포인트** — 요청/응답 처리, 에러 핸들링
- **유틸리티 함수** — 순수 함수, 변환 로직, 파싱
- **버그 수정** — 재현 테스트를 먼저 작성하고 수정

### TDD 불필요 대상

- **타입 정의** — TypeScript interface, type alias
- **설정 파일** — `vite.config.ts`, `tsconfig.json`, `.eslintrc`
- **순수 UI 컴포넌트** — 로직 없는 렌더링 전용 컴포넌트
- **SQL/마이그레이션** — 스키마 정의, 데이터 마이그레이션

---

## 2. RED → GREEN → REFACTOR 사이클

### RED — 실패하는 테스트 작성

- **하나의 동작 = 하나의 테스트.** 여러 동작을 한 테스트에 묶지 마라.
- **반드시 테스트를 실행해서 실패를 눈으로 확인**하라. 실행 없이 다음 단계로 넘어가는 것은 금지.
- 실패 이유가 **"기능 미구현"**이어야 한다. `SyntaxError`나 `ImportError`로 실패하면 테스트 코드 자체가 잘못된 것이다.

### GREEN — 테스트를 통과시키는 최소 코드 작성

- **최소한의 코드만** 작성하라. 테스트를 통과시키는 것 이상을 구현하지 마라.
- **YAGNI** — 지금 필요하지 않은 기능은 추가하지 마라.
- 신규 테스트와 **기존 테스트 전체가 통과**하는지 확인하라.

### REFACTOR — 동작을 유지하며 코드 개선

- **중복 제거, 이름 개선, 헬퍼 추출**만 허용한다.
- 리팩터 중 **green 상태를 유지**하라. 테스트가 깨지면 즉시 롤백.
- **새로운 동작을 추가하지 마라.** 다음 RED 단계에서 처리하라.

### 반복

리팩터 완료 후 다음 동작에 대한 **RED로 돌아가라.**

---

## 3. 삭제 강제 규칙

- 테스트 없이 프로덕션 코드를 먼저 작성했다면 **즉시 삭제하고 RED부터 재시작**하라.
- "나중에 테스트 추가하겠다"는 없다. 지금 삭제하라.
- **"참고용"으로 남기는 것도 금지**한다. 테스트 없는 코드는 검증되지 않은 코드다.

---

## 4. 변명 차단표

| 변명 | 반론 |
|------|------|
| "너무 단순해서 테스트 불필요" | 단순한 코드도 회귀한다. 단순하면 테스트 작성도 30초다. |
| "나중에 추가하겠다" | 나중은 오지 않는다. 기술 부채는 복리로 쌓인다. |
| "시간이 없다" | 테스트 없이 디버깅하는 시간이 더 길다. 지금 투자가 이후를 단축한다. |
| "삭제하면 낭비다" | 검증 안 된 코드를 남기는 게 더 큰 낭비다. 삭제는 손실이 아닌 정직함이다. |
| "프로토타입이다" | 프로토타입은 버린다. 프로덕션에 올라가면 프로토타입이 아니다. |
